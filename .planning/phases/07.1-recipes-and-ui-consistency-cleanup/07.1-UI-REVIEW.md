## Delta vs. Phase 7 Review

Phase 7 scored **Typography 2/4** and **Spacing 2/4** as its two weakest pillars and named them as Top Fixes #1 and #2. Phase 7.1 explicitly targeted both:

- **Typography: 2/4 → 3/4.** The fix landed as specified — six tokens, 20+ selectors migrated (26 measured), 0.68rem floor eliminated. This is not a 4/4 because the plan's own scope note admits it: "Do NOT touch the pre-Phase-7 px-based font sizes... out of scope." Grepping the current file shows **113 raw, un-tokenized `font-size` declarations** still outnumber the 26 tokenized ones roughly 4:1, and several selectors set literal values that exactly equal a token (e.g. `font-size: 1.05rem;` on one selector vs. `font-size: var(--fs-lg);` on another, both meaning "1.05rem") — meaning the scale is additive, not yet canonical. One full step of credit for a real, verified fix; not two steps, because the underlying sprawl this pillar measures is still the majority case in the file.
- **Spacing: 2/4 → 3/4.** Same shape of result. The four named classes the Phase 7 review prescribed by name exist verbatim and are wired into both target files with real call sites (not just defined-and-unused). Inline count dropped by a third in each file (recipes.js 87→55, settings.js 38→24) — both drops match the plan's own stated thresholds (<60, <25) but those thresholds were deliberately set as partial-progress markers, not "done" markers. The CLAUDE.md convention ("all visual decisions live in app.css") is still violated by the majority of remaining inline styles, and the review found at least one instance (`recipes.js:1520`) where the newly-built `.muted-help` class was not applied to a pattern immediately adjacent to code that WAS migrated — evidence the pass was pattern-driven grep-and-replace rather than an exhaustive audit.

Both deltas are genuine improvements verified against runtime-measured facts (fs-lg = 15.75px, fs-xs = 11.25px, space-3 = 12px) and are not inflated — but neither pillar should be scored 4/4 given the remaining sprawl documented above. Scoring 3/4 for both is the fair reflection of "meaningfully better, not solved."

## Top 3 Priority Fixes (Remaining)

1. **Finish the font-size token migration outside the originally-scoped selector set.** 113 raw `font-size` declarations remain in `app/css/app.css` vs. 26 tokenized ones. Concrete quick win: `font-size: 0.82rem;` appears 17 times (lines 153, 258, 363, 406, 407, 445, 476, 487, 526, 755, 791, 807, 815, 830, 870, 909, 1040) — none of these reference `var(--fs-sm)` even though 0.82rem is closer to `--fs-sm` (0.85rem) than any other step. Snapping just this one repeated value to `var(--fs-sm)` removes 17 raw declarations in one pass. Impact: pushes the tokenized ratio from ~19% to ~34% of all font-size declarations with a single find-and-replace.

2. **Close the inline-style gap inside already-migrated form-section-card blocks.** `app/js/views/recipes.js:1520` (`Generate with AI` helper paragraph, inside `#rf-ai-wrap.form-section-card`) and `:1500`/`:1503`/`:1506` (`Tweak with AI` panel, inside `#rf-tweak-wrap.form-section-card`) still hand-write `style="font-size:0.82rem;color:var(--text-dim);..."` — the literal pattern `.muted-help` exists to replace, one DOM level below a wrapper that WAS converted. Apply `class="muted-help"` (dropping the inline font-size/color) to these four sites. Impact: removes 4 of the 55 remaining inline styles in the exact surfaces this phase's own plan named as targets, and eliminates a visible inconsistency where sibling elements in the same form block use two different styling mechanisms for the identical visual pattern.

3. **Add `aria-label` to the four icon-only chip action buttons.** `app/js/recipe-chip.js:162-171` (`favorite`/`wishlist`/`made`/`discard`) render `<button class="btn-icon" data-action="..." title="...">` with a Unicode glyph as the only visible content and no `aria-label`. This is unchanged from the Phase 7 review's Visuals finding and is now MORE consequential because CHIP-04 made these same buttons appear on every Originals chip too — the surface area of the gap grew with this phase's own change. Fix: add `aria-label="${title text}"` matching each existing `title=` value in `_actionsHtml()`. Impact: screen-reader users get equivalent affordance on a UI element whose reach just expanded from 4 tabs to 5.

### Pillar 4: Typography (3/4) — up from Phase 7's 2/4

**What changed (verified):**
- Six-step scale added to `:root` (`app/css/app.css:22-28`): `--fs-xs: 0.75rem`, `--fs-sm: 0.85rem`, `--fs-base: 0.95rem`, `--fs-lg: 1.05rem`, `--fs-xl: 1.2rem`, `--fs-2xl: 1.5rem`.
- 26 selectors now reference `var(--fs-*)` (`grep -c 'font-size: var(--fs-' app/css/app.css` → 26), including every selector the Phase 7 review named by exact line: `.recipe-chip-name` → `var(--fs-lg)`, `.recipe-chip-meta` → `var(--fs-sm)`, `.badge-status` → `var(--fs-xs)`, `.chat-bubble` → `var(--fs-base)`, `.lesson-tile-title` → `var(--fs-lg)` (spot-checked at `app.css:1879-1936`).
- Runtime-measured confirmation (headless computed styles): `.badge-status` computes to 11.25px = 0.75rem × 15px base — the 0.68rem accessibility floor Phase 7 flagged is gone, replaced by a value above the 12px-equivalent target once base font-size (15px) is accounted for. `.recipe-chip-name` computes to 15.75px = 1.05rem × 15px, matching `--fs-lg`.

**What did not change (why this isn't a 4/4):**
- `grep -oE 'font-size: *[0-9.]+(rem|px);' app/css/app.css | wc -l` → **113** raw, non-token font-size declarations remain, nearly 4.3× the 26 tokenized ones. Top offenders by frequency: `0.82rem` (17×), `12px` (11×), `0.88rem` (10×), `0.9rem` (9×), `0.75rem` as a raw literal rather than `var(--fs-xs)` (9×), `0.78rem` (8×), `0.85rem` as a raw literal rather than `var(--fs-sm)` (7×).
- Several selectors duplicate a token's exact numeric value without referencing the token: `font-size: 0.75rem;` appears 9 times as a literal alongside 4 uses of `var(--fs-xs)` (also 0.75rem); `font-size: 0.85rem;` appears 7 times as a literal alongside 7 uses of `var(--fs-sm)`; `font-size: 0.95rem;` appears 3 times as a literal alongside 9 uses of `var(--fs-base)`; `font-size: 1.05rem;` appears 2× as a literal alongside 4 uses of `var(--fs-lg)`; `font-size: 1.2rem;` appears 1× literal alongside 1 use of `var(--fs-xl)`; `font-size: 1.5rem;` appears 4× literal alongside 1 use of `var(--fs-2xl)`. This means a future change to any single token will NOT propagate to roughly half the places that currently render at that exact size — the scale is real but not yet canonical.
- The plan's own scope note explicitly excluded the pre-Phase-7 px-based selectors (~lines 1062-1409, flow-chart styles) — that mixed-unit sprawl (`12px`, `13px`, `14px`, `20px` all appear as raw px, 18 occurrences combined) is untouched and still coexists with the new rem-based token scale.

**Verdict:** genuine, verified progress (2/4 → 3/4) on exactly the selectors named in the Phase 7 fix; not a 4/4 because the majority of font-size declarations in the file are still ad-hoc and the token scale has not yet displaced its own duplicate literal values.

### Pillar 5: Spacing (3/4) — up from Phase 7's 2/4

**What changed (verified):**
- Six-step spacing scale added to `:root` (`app/css/app.css:29-35`): `--space-1: 4px` through `--space-6: 24px`.
- The four classes the Phase 7 review prescribed by name exist and reference the tokens (`app.css:269-304`): `.form-section-card` (padding/margin/background/border all tokenized), `.form-actions-row` (flex/gap/margin tokenized), `.muted-help` (font-size/color tokenized), `.input-disabled` (opacity/cursor — intentionally not spacing-tokenized, correct as-is). - These classes are demonstrably wired into real call sites, not merely defined-and-unused: `recipes.js:1497` and `:1518` use `form-section-card`; `:1616` uses `form-actions-row`; `:1603`/`:1607` use `form-check-row`; `:1611` uses `input-disabled`; `settings.js:133/166/194/208/276` use `form-status`; `settings.js:190/205/212/238` use `muted-help` (4 occurrences, meeting the plan's own ≥4 acceptance threshold).
- Inline `style=` counts dropped as claimed: `recipes.js` 87→55 (verified via `grep -o 'style=' | wc -l`), `settings.js` 38→24 (verified) — both below the plan's own stated thresholds (<60, <25).

**What did not change (why this isn't a 4/4):**
- 55 and 24 inline styles respectively are still substantial counts — this was a partial extraction of the highest-frequency repeated patterns, not an elimination. The CLAUDE.md convention ("all visual decisions live in `app/css/app.css`") is still violated by the majority of remaining cases.
- Concrete evidence of incomplete/inconsistent application within the SAME migrated block: `recipes.js:1497-1507` wraps the "Tweak with AI" panel in `class="rf-gen-wrap form-section-card"` (migrated), but three lines inside that same wrapper still hand-write `style="font-size:0.82rem;..."` (`:1500`, `:1503`, `:1506`) — the exact `.muted-help` pattern, unapplied one DOM level below a converted parent. The same shape repeats at `:1518-1524` (`#rf-ai-wrap`, migrated wrapper) with `:1520`'s paragraph still inline-styled with the identical `font-size:0.82rem;color:var(--text-dim);margin-bottom:10px;` pattern `.muted-help` was purpose-built to replace.
- The disabled-checkbox pattern (`recipes.js:1608-1613`) is now split across two mechanisms: the `<input>` gets `class="input-disabled"` (correct), but the sibling `<label>` still uses `style="margin:0;${r?.seed_id ? 'opacity:0.6;' : ''}"` — a conditional inline style duplicating what the class mechanism was introduced to replace, on the adjacent element in the same disabled-state UI unit.

**Verdict:** genuine, verified progress (2/4 → 3/4): real tokens, real named classes, real reduction, real usage — but the remaining inline counts are still large relative to the file sizes, and the review found direct evidence of missed opportunities immediately adjacent to converted code, indicating the pass was a targeted grep-driven extraction rather than an exhaustive sweep.

No new gaps found in this pillar for this phase's scope; nothing in 07.1 touched loading/error/empty-state coverage outside what's already covered above under Copywriting/Visuals, and no destructive-action or key-gating regressions were found in the diffed surfaces.
