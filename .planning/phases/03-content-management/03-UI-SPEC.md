# Phase 3 — UI Design Contract

> Visual and interaction contract for the Content Management phase. Generated by gsd-ui-researcher.
> All values reuse existing CSS custom properties from `app/css/app.css`. No new tokens are introduced.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla JS SPA, no build step, no framework) |
| Preset | not applicable |
| Component library | none — custom CSS in single stylesheet `app/css/app.css` |
| Icon library | inline SVG (existing pattern in `index.html`) + a small set of emoji glyphs already used in the codebase (`✨`, `✕`, `⏳`, `🍹`, `📋`, `⭐`) |
| Font | `'Georgia', serif` at 15px base / 1.6 line-height (declared on `html, body`) |

Architecture constraints (locked):
- IIFE module pattern (`const FooView = (() => { return { render }; })();`)
- Views render into `#main-content` via a hash router
- No new routes for this phase (D-01): recipe form replaces view content in-place
- No external libraries except JSZip (loaded via `<script>` from CDN in `index.html`) for ZIP pack/unpack
- All new styles append to `app/css/app.css` — no separate stylesheet

Existing palette (CSS custom properties on `:root`) — reused unchanged. The 60/30/10 split below maps to actually-rendered surface area.

| Role | Token | Value | Usage in Phase 3 |
|------|-------|-------|------------------|
| Dominant (60%) | `--bg` | `#0f0f0f` | Page background |
| Secondary (30%) | `--bg2` | `#1a1a1a` | Settings section cards, form `<wrap>` containers, recipe detail card surface |
| Surface raise (within 30%) | `--bg3` | `#242424` | Form inputs, ingredient row strip, drop-zone idle state, import preview list rows, AI prompt textarea |
| Surface raise level 2 | `--bg4` | `#2e2e2e` | Hover state on secondary buttons; coming-soon badge background (reused unchanged) |
| Accent (10%) | `--amber` | `#d4943a` | Primary CTA backgrounds, focus rings on inputs, drop-zone active border, recipe form H2, Generate-with-AI button accent edge |
| Accent dim | `--amber-dim` | `#a06e22` | Section labels, hover border on chips and import preview rows |
| Accent glow | `--amber-glow` | `#f0b860` | `.btn-primary:hover` (existing) |
| Destructive | `--red` | `#e05555` | Save validation errors, upload failure status, Delete button, "Reset all data" |
| Success | `--green` | `#4caf7d` | Upload success status, import success toast border, "Confirm Import" success state |
| Info | `--blue` | `#5b9bd5` | Info toast border only — not used for any Phase 3 control |
| Text | `--text` | `#e8e2d9` | Default text |
| Text dim | `--text-dim` | `#9a9080` | Helper text, labels, placeholder-adjacent labels |
| Text muted | `--text-muted` | `#666` | Status idle text, placeholder text |
| Border | `--border` | `#333` | Card borders, form input borders |
| Border raise | `--border2` | `#444` | Input border, hover border on ghost buttons |

**Accent (`--amber`) is reserved exclusively for:**
1. The primary CTA button background (`.btn-primary`) — Save, Confirm Import, Create Recipe, Generate
2. Focus state of form inputs (`input:focus`, `textarea:focus`, `select:focus`)
3. The active drop-zone border (drag-over state)
4. Page H1 and form H2 text color
5. The recipe ID monospace label in detail view (existing pattern)
6. Active tab underline (existing `.tab.active`)

**Accent is NOT used for:** secondary buttons, ghost buttons, plain links inside body text, or non-CTA chips. This preserves the 60/30/10 dominance.

**Destructive (`--red`) is reserved exclusively for:**
1. Delete recipe button (existing)
2. "Reset all data" confirm button (existing)
3. Validation error text (`Name is required`, `Creator is required`, missing required fields)
4. Upload / save / import failure status lines
5. The "Danger Zone" settings section border

## Responsive Contract

The existing breakpoint at `max-width: 640px` is the only breakpoint this phase respects. Phase 3 additions:

- Recipe form: `<wrap>` `max-width: 680px` is preserved; on mobile the `.form-row` collapses to a column via existing media-query rule (line 597).
- Settings Export section: button row `flex-wrap: wrap` ensures Download ZIP and Export for AI stack on narrow screens.
- Drop zone: full-width on all viewports; `padding: 24px` reduces to `padding: 16px` at `max-width: 640px`.
- Import preview dialog: `width: 90%` (inherits existing `.confirm-dialog` rule) — works on mobile.
- AI Generate block on mobile: textarea remains 100% width; Generate button takes full width (`.btn-block`-equivalent inline style) on `max-width: 480px`.

## Accessibility Floor

- All form inputs have associated `<label>` elements (existing pattern preserved).
- Buttons that change state (Generate, Save, Upload, Confirm Import) reflect their state in text (not only color/spinner). Screen readers will announce label changes.
- Drop zone has a fallback click-to-pick path (`<input type="file">`) so keyboard-only users can use it.
- Show/Hide API key toggle changes both the input `type` and the button label, so keyboard users get equivalent affordance.
- Color is never the sole signal for state (status text accompanies every red/green color change in upload, generate, and import flows).
- No `window.confirm()` for destructive actions — use the existing `.confirm-dialog` overlay pattern (matches Phase 2 logout dialog). Note: the existing `Delete` button in recipes.js still uses `confirm()` — Phase 3 does NOT change this behavior (out of scope; deferred recipe-deletion work is noted in CONTEXT.md deferred section).
