## 1. System Classification

**System Type:** Hybrid — Conversational Assistant + Content Generation (single-model chat with structured-JSON generation; no autonomous tool-use loop)

**Description:**
A browser-side, bring-your-own-key (BYOK) AI assistant embedded in a static no-backend SPA. It impersonates the user's configured home-bartender persona ("Bjorn") and serves two conversational surfaces (a persisted `#chat` page and an ephemeral quick-ask drawer) plus a set of one-shot generation features: AI cocktail design, AI-powered recommendation explanations, inventory advice, lesson-scoped Classroom Q&A, and AI-assisted data parsing/repair (legacy markdown import, JSON error correction, paste-a-line fallback, persona drafting, ingredient-derivation inference). All calls go directly from the browser to the Anthropic Messages API. "Good" means responses sound like a knowledgeable bartender with full, accurate context of the user's bar/tastes/history, generated recipes are buildable from the stated inventory and conform to the universal recipe JSON schema, and structured-output features never silently corrupt the user's GitHub-stored data.

**Critical Failure Modes:**
1. **Silent data corruption** — an AI-parsed/repaired/generated payload (import, JSON repair, recipe generation) is written to GitHub without a diff preview + explicit user confirmation, or violates `schema/*.json`, overwriting good user data.
2. **Schema-invalid generation** — a generated recipe or extracted object omits required fields or uses the wrong shape, so it cannot render as a chip or breaks the recommender engine.
3. **Context infidelity / hallucinated inventory** — Bjorn recommends or "designs" drinks using spirits the user does not own (ignoring inventory + vetoes), or contradicts the locked flavor profile/persona.
4. **Key/credential exposure** — the Anthropic key is logged to the call log, committed to GitHub, sent anywhere other than `api.anthropic.com`, or rendered unmasked; or the `anthropic-dangerous-direct-browser-access` header is omitted so calls silently fail.
5. **Unbounded cost / runaway calls** — missing prompt caching, no abort on navigation, or retry storms on 429 drive surprise spend on the user's own key.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Consumer hobby — food & beverage / craft cocktails & mixology education. Not a regulated vertical (not healthcare, legal, or finance); the closest analogs are recipe-recommendation and hobbyist-instruction apps.

**User Population:** Home-bar enthusiasts and craft-cocktail hobbyists running a personal, self-hosted browser app with their own Anthropic (BYOK) key and their own data stored in their own GitHub repo. Single-user, single-tenant: each person is owner, operator, and audience of their own bar data. Skill ranges from curious beginner to serious amateur mixologist.

**Stakes Level:** **Low overall** (a hobby app, no commercial/clinical/financial consequence), with two genuine risk vectors that elevate specific outputs to **Medium**: (1) **Consumable safety** — the output is a drink the user actually mixes and ingests, so responsible-consumption framing, allergen awareness (egg white, dairy, nuts, citrus), and unsafe practices (dangerous infusions, flaming, unwarned overproof/quantity) matter; (2) **Data integrity** — AI-parsed/generated content is written to the user's GitHub data files, where a bad write can silently corrupt real records.

**Output Consequence:** The user reads, trusts, and acts on Bjorn's output by **mixing and drinking the suggested cocktail** and/or by **committing AI-generated/parsed content to their own GitHub data files** (recipes, drafts, inventory, profile). A wrong recipe wastes ingredients and yields an undrinkable or unsafe drink; a wrong write corrupts the user's system of record.

### What Domain Experts Evaluate Against

A professional bartender / experienced mixologist judges a recommendation or AI-designed drink against the following — in bartender language, not AI jargon.

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|--------------------|--------|--------|
| **Cocktail balance** (sweet–sour–strong–bitter–dilution) | No single axis dominates; sweetness softens acid/bitterness, dilution lands a drink at ~20–25% water by volume; the drink is *drinkable*. | Cloying, harsh, or flabby — e.g. all spirit with no counterweight, or sugar with no acid to cut it; an undrinkable combination. | Medium | Mixology balance theory (sweet/sour/bitter/strong + dilution as the five levers). |
| **Technique correctness** | Shake drinks with citrus, juice, dairy, or egg (aeration + chilling, dry-shake egg first); stir spirit-only/spirit-forward drinks for clarity and tight dilution; correct ice and chill. | Recommends *shaking* a Martini/Manhattan/Negroni (cloudy, over-diluted) or *stirring* a sour/egg drink (flat, unemulsified, separated). | Medium | Bartender shake-vs-stir golden rule. |
| **Ratio sanity** | Recognizable specs hold: a sour ≈ 2:1:1 (spirit:sour:sweet), a Negroni ≈ 1:1:1 (gin:sweet vermouth:Campari), an Old Fashioned is spirit-forward (spirit + small sweetener + bitters). | Off-template ratios that no bartender would pour — e.g. a "sour" that is mostly syrup, or an "Old Fashioned" diluted with juice. | Medium | Classic cocktail ratio conventions. |
| **Glassware & garnish appropriateness** | Vessel and garnish fit the drink (coupe/Nick & Nora for stirred-up, rocks for spirit-forward on ice, highball for long drinks; citrus twist expresses oil, mint slapped not bruised). | Wrong vessel (sour served neat in a rocks glass with no logic) or a garnish that fights the drink (lime on a stirred whiskey drink "because"). | Low | Standard service conventions. |
| **Ingredient compatibility** | Flavor pairings make sense (agave + lime + orange liqueur; whiskey + bitters + sugar); modifiers complement the base spirit and the owner's flavor profile. | Clashing or incoherent combinations (e.g. cream + citrus that will curdle, competing strong amari with no anchor). | Medium | Flavor-pairing / structure fundamentals. |
| **Buildability from on-hand inventory** | Every ingredient is in the user's actual inventory; vetoes are honored; substitutions are flagged as substitutions, not invented as owned. | "Phantom bottles" — uses spirits/liqueurs the user does not own, or uses a veto'd/disliked ingredient. (Maps to Section 1 Critical Failure Mode #3.) | High | Core product promise: drinks must be buildable from *this* bar. |
| **Naming & originality** (AI-designed drinks) | An original drink gets a fitting, non-duplicative name; provenance of any *classic* cited is accurate or omitted, not invented. | Fabricated history/provenance for a "classic," or a "new" drink that is just a renamed standard presented as original. | Low–Medium | Bartender honesty norm + hallucination control. |

### Known Failure Modes in This Domain

How generic LLM failure manifests specifically in mixology:

1. **Unbalanced / undrinkable combo** — a recipe that technically lists ingredients but no bartender would serve (no acid to cut the sugar, all-spirit with no dilution path, clashing modifiers).
2. **Wrong technique for the build** — instructs "shake" on a stirred spirit-forward drink (cloudy, watery) or "stir" on a citrus/egg drink (flat, separated, no foam).
3. **Phantom-ingredient recipe** — designs or recommends a drink calling for bottles the user does not own, or ignores the inventory/veto list entirely (ties to Critical Failure Mode #3).
4. **Fabricated provenance** — invents a confident-sounding "classic" origin story, inventor, or year for a cocktail.
5. **Ignored vetoes / allergens** — recommends an ingredient the user vetoed or is allergic to (egg white, dairy, nuts, citrus) without flagging it.
6. **Unsafe quantity / overproof without warning** — large pours, overproof spirits, or risky prep (dangerous home infusions, flaming) presented with no responsible-consumption caveat.
7. **Generic, non-personalized suggestion** — a "default" crowd-pleaser that ignores the locked flavor profile, mood, and made-history, so it doesn't sound like *the user's* bartender.

### Regulatory / Compliance Context

**No legally binding regulatory regime applies** to this deployment: it is a single-user, self-hosted hobby app with BYOK credentials and the user's own data in their own repo — not a regulated SaaS, not a vendor selling alcohol, not handling third-party PII at scale. There is no HIPAA/FCA/GDPR-controller obligation created by the system itself.

The following are **recommended editorial guardrails** (soft constraints the persona/output *should* honor), not compliance mandates:
- **Of-age framing** — assume an adult (21+ US / legal drinking age) audience; do not encourage underage drinking.
- **Responsible-consumption messaging** — favor balanced, single-serving suggestions; surface a light caveat on overproof or high-volume builds.
- **Allergen disclosure** — call out common allergens (egg white, dairy, nuts) and honor user vetoes/allergies.
- **No medical/health claims** — never frame alcohol as healthful or therapeutic; no dosing/health advice.
- **User-data privacy** — the BYOK Anthropic key and all bar data stay in the user's browser/own repo; the AI must never log or exfiltrate the key (ties to Critical Failure Mode #4).

**Version:** Anthropic API version header `2023-06-01`. No package dependency (zero-build project). Prompt caching via `cache_control` beta is GA on `2023-06-01` and requires no separate beta header. Model IDs resolved at call time via the SET-05 selector (`localStorage.bb_chat_model`); default and selectable IDs to be confirmed against current Anthropic model docs by `gsd-ai-researcher` (Section 4).

**Rationale:**
The project is a deliberately zero-dependency, no-build, no-backend static SPA (vanilla ES6 IIFE modules — see `CLAUDE.md`). It is committed to Anthropic-only, BYOK, browser-direct calls, and `claude-api.js` already issues a non-streaming Messages API call via `fetch`. The decision matrix's Anti-Pattern #1 ("Using LangChain for simple chatbots — a direct SDK call is less code, faster, and easier to debug") applies directly: this is a single-model conversational + content-generation system with no retrieval store, no multi-agent orchestration, and no autonomous tool loop, so an agent framework adds only abstraction and a mandatory build step. User confirmed (2026-05-21) the direct-`fetch` approach over adopting the official SDK.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| `@anthropic-ai/sdk` (official TS SDK) | Cleaner streaming/caching ergonomics, but requires introducing npm + a bundler/build step — a major regression of the project's no-build ethos. User explicitly declined. |
| Vercel AI SDK | Designed around a Node/Next.js server route to hide the key; this app has no backend and uses BYOK browser-direct. |
| Claude Agent SDK | For code-aware autonomous/tool-loop agents; this system has no autonomous loop and is browser-side. Overkill. |
| LangChain / LangGraph / LlamaIndex | Python-first (or heavy TS) agent/RAG frameworks; no retrieval store or graph workflow here, and all require a build toolchain. |

**Vendor Lock-In Accepted:** Yes (conscious) — Anthropic-only is a locked product decision; BYOK keeps the cost/credential with the user. No framework lock-in (raw HTTP is portable).

### Entry Point Pattern — streaming chat via fetch + ReadableStream (SSE)
```javascript
// Robust SSE reader: buffers partial lines across chunk boundaries (a common bug).
async function streamMessage(body, { onText, signal }) {
  const resp = await fetch('https://api.anthropic.com/v1/messages', {
    method: 'POST',
    headers,                                  // includes the browser-direct header
    body: JSON.stringify({ ...body, stream: true }),
    signal,                                   // AbortController.signal (CHAT-06)
  });
  if (resp.status === 429) {
    const retryAfter = resp.headers.get('retry-after');   // CHAT-09
    throw new RateLimitError(retryAfter);
  }
  if (!resp.ok) throw new Error(`Anthropic ${resp.status}: ${await resp.text()}`);

  const reader = resp.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '';
  let usage = null;
  for (;;) {
    const { value, done } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop();                       // keep the last partial line
    for (const line of lines) {
      if (!line.startsWith('data:')) continue;  // ignore `event:` and blank lines
      const data = line.slice(5).trim();
      if (!data) continue;
      const evt = JSON.parse(data);
      if (evt.type === 'content_block_delta' && evt.delta.type === 'text_delta') {
        onText(evt.delta.text);                 // append token to the transcript
      } else if (evt.type === 'message_delta') {
        usage = evt.usage;                       // output tokens, etc. → AI-09 log
      } else if (evt.type === 'error') {
        throw new Error(evt.error?.message || 'stream error');  // CHAT-08
      }
      // message_start / content_block_start/stop / message_stop / ping: ignore
    }
  }
  return usage;
}
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `system` (array of text blocks) | Persona + bar context, separate from the chat turns | Put the big stable context here; attach `cache_control` to its last block |
| `cache_control: {type:"ephemeral"}` | Prompt-cache breakpoint (≤4 per request) | On the last stable `system` block so the rich context is reused across turns (D-05) |
| `messages` (user/assistant array) | The conversation; stateless — full history sent each call | Running summary + recent turns + current message |
| SSE events | `message_start` → `content_block_start` → `content_block_delta` (`text_delta`) → `content_block_stop` → `message_delta` (carries `stop_reason`+`usage`) → `message_stop`; plus `ping` and `error` | Parse `content_block_delta.text_delta` for tokens; `message_delta.usage` for the log; `error` for CHAT-08 |
| `AbortController` | Cancels the in-flight `fetch` | Tie to view lifecycle; `cleanup()` calls `.abort()` (CHAT-06) |
| `max_tokens` | Hard per-response output cap (model unaware) | Bound output length + cost on the user's key |

### Common Pitfalls
1. **Missing `anthropic-dangerous-direct-browser-access: true`** → the browser request is blocked (CORS preflight) / 401. It is mandatory for in-browser calls.
2. **SSE lines split across chunks** — a `data:` JSON object can span two `reader.read()` chunks. Buffer and split on `\n`, keep the trailing partial (shown above). Naive `JSON.parse` per chunk throws intermittently.
3. **Prompt-cache minimum prefix** — caching only kicks in above the model minimum: **4096 tokens** for Opus 4.7 / Opus 4.6 / Haiku 4.5, **2048 tokens** for Sonnet 4.6. A smaller context silently won't cache (`cache_creation_input_tokens: 0`, no error). The rich D-04 context comfortably clears this.
4. **Cache TTL** — ephemeral cache is ~5 min by default (`{type:"ephemeral","ttl":"1h"}` for 1h). Don't expect cross-session reuse; re-warm on data change, not every keystroke.
5. **Volatile data inside the cached block busts the cache every turn** — never interpolate a timestamp/UUID/the current question into the cached `system` block. Keep the question in `messages` (D-05 / prompt-caching prefix rule).
6. **Sampling params are model-specific** — `temperature`/`top_p`/`top_k` are rejected (400) on Opus 4.7; `effort` errors on Haiku 4.5. With a Haiku/Sonnet/Opus selector (SET-05), the safest cross-model request omits all of them (see §4).
7. **429 retry storms** — surface `retry-after`; do NOT auto-retry in a loop (cost on the user's key).
8. **Key hygiene** — the key only ever goes in the `x-api-key` header to `api.anthropic.com`; never log it to `bb_api_log`, never commit it.

**Per-feature parameters:**
- `max_tokens`: chat ≈ 1024–2048; recipe design ≈ 1024; JSON repair/import ≈ 2048–4096 (output can be large); paste-a-line/derivation ≈ 256–512. A cap is mandatory (bounds cost — FM #5).
- **Sampling / thinking — keep the request minimal and model-agnostic.** Because the SET-05 selector spans Haiku/Sonnet/Opus and the constraints differ by tier (Opus 4.7 rejects `temperature`/`top_p`/`top_k` with a 400; `effort` errors on Haiku 4.5; Opus 4.7 thinking is adaptive-only), **do NOT send `temperature`, `top_p`, `top_k`, `thinking`, or `effort`** in the base request. Control behavior through the system prompt instead. This keeps one request shape valid across all three tiers and avoids per-model branching. (If a future feature needs deeper reasoning, gate `thinking`/`effort` on the selected tier — but the v1 chat/generation features do not need it, and omitting it is cheaper and lower-latency.)

**Core Pattern:**
- **Streaming chat turn** (`#chat` + drawer): build `system` = [cached context block(s)] (D-04/D-05), `messages` = [running-summary text, …recent turns, current user message], `stream: true`; render `text_delta` tokens into the transcript live (the `streamMessage()` pattern in §3).
- **One-shot structured generation** (recipe design, import, repair, derivation, persona draft): `stream: false`, JSON-only system instruction, parse + validate + (one retry) — see §4b `requestJSON()`. Then show a diff/preview and require explicit confirm before any GitHub write.

**Tool Use:**
None. This system has **no function calling / tool use and no server-side tools**. Structured output is achieved by JSON-only prompting + client-side validation against `schema/*.json` (and the `Normalize` module), not by the tool API.

**State Management:**
- `#chat` thread persisted to `localStorage.bb_chat_history` (D-03); cleared only by explicit user action; resumes on return. Quick-ask drawer is in-memory only (ephemeral, resets each open).
- "Save conversation to GitHub" (manual, D-03) and all AI-generated data (`data/drafts.json`, import/repair writes) go through the existing `GitHubAPI.writeJSON()` (base64 + SHA) path — never an automatic per-message commit.
- The Anthropic key lives only in `localStorage.bb_anthropic_key`; the AI-09 call log (`localStorage.bb_api_log`, 50-entry cap) records timestamp/type/model/token-usage but **never the key**.

**Context Window Strategy:**
Per D-04/D-05/D-07: a large stable **cached** `system` block (persona + flavor profile + mood + originals + drafts + made-log/tallies + inventory/vetoes, rendered as derived markdown — D-06) reused via `cache_control`; older turns condensed into a **running summary** sent verbatim; the last N turns sent in full; the current message last. Watch `usage.cache_read_input_tokens` to confirm the cache is hitting (D-05). The 200K–1M context windows are far larger than this app needs — caching is for cost/latency, not capacity.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs (Pydantic N/A — JS equivalent)

Pydantic is **not applicable** (no Python). The browser-JS equivalent is a `requestJSON()` helper that enforces JSON-only output, parses it, validates against the relevant `schema/*.json` (and/or the project's `Normalize` module), retries **once** on failure, and fails closed (no write) on a second failure. This is the structured-output contract for every generation/parse feature (AI-03/08/10/11/13).

```javascript
// Structured-output contract: JSON-only → parse → validate → one retry → fail closed.
// Note: the public Messages API also offers `output_config.format` (JSON schema) on
// current models; for a zero-dependency browser app the prompt-and-validate approach
// below is simplest and keeps validation authoritative against the app's own schema/*.json.
async function requestJSON({ system, userPrompt, schemaKey, model, maxTokens }) {
  const ask = (extra = '') => callMessages({                 // non-streaming Messages call
    model, max_tokens: maxTokens,
    system: [{ type: 'text', text: system + '\nRespond with a SINGLE JSON object and nothing else — no prose, no markdown, no code fences.' }],
    messages: [{ role: 'user', content: userPrompt + extra }],
  });

  let raw = await ask();
  for (let attempt = 0; attempt < 2; attempt++) {
    let obj;
    try { obj = JSON.parse(stripFences(raw)); }              // tolerate stray fences
    catch (e) { obj = null; }
    if (obj) {
      const norm = Normalize.byKey(schemaKey, obj);          // coerce to canonical shape
      const errors = validateAgainstSchema(schemaKey, norm); // checks schema/<key>.schema.json
      if (errors.length === 0) return norm;                  // success
      if (attempt === 0) { raw = await ask(`\n\nYour previous output was invalid: ${errors.join('; ')}. Return corrected JSON only.`); continue; }
      throw new StructuredOutputError(schemaKey, errors);    // fail closed — caller must NOT write
    }
    if (attempt === 0) { raw = await ask('\n\nThat was not valid JSON. Return a single valid JSON object only.'); continue; }
    throw new StructuredOutputError(schemaKey, ['unparseable JSON']);
  }
}
```
The caller then renders a **diff/preview** and requires explicit user confirmation before persisting (FM #1). Never write a payload that failed validation.

### Async-First Design

The browser is async by default. Chat **streams** (`fetch` + `ReadableStream` — §3); one-shot JSON generation **awaits** the full response. The single most common mistake: not wiring an `AbortController` to the view lifecycle, so navigating away leaks an in-flight request and burns the user's tokens. Every surface creates a controller on open and calls `.abort()` in its `cleanup()` (CHAT-06). The persisted transcript (D-03) is unaffected by an abort.

### Prompt Engineering Discipline

- **System vs. user separation:** the rich, stable bar context (persona/profile/mood/inventory/originals/drafts/made-log) is the `system` block (cached); the user's question/instruction is the `messages` turn. Never mix volatile content into the cached `system` block (busts the cache — §3 pitfall 5/6).
- **JSON-only discipline** for generation/parse features: explicit "single JSON object, no prose/markdown/fences" instruction, reinforced by the one-retry `requestJSON()` contract.
- **Token budget:** cache the big block once; keep per-turn deltas (summary + recent turns + question) small. Generated recipes must conform to the universal recipe JSON schema so they render as chips everywhere (Phase 6 D-02).

### Context Window Management

Conversation summarization (D-07): when the persisted `#chat` thread grows, condense older turns into a running summary (one extra cheap call, e.g. Haiku) and send that verbatim ahead of the last N full turns. Keep the full transcript locally for display only. The drawer never accumulates (ephemeral). No server-side compaction is used (browser-only, no backend).

### Cost and Latency Budget

- Cost is on the **user's BYOK key**, so frugality is a product virtue, not just ops hygiene.
- **Prompt-cache hit rate is the main lever** (D-05): cache reads cost ~0.1× input; a stable cached context block turns a large per-turn context into a near-free prefix. Verify via `usage.cache_read_input_tokens`; if it's persistently 0, a silent invalidator is in the cached block (§3 pitfall 5).
- **Model routing by task** (SET-05 + per-feature default): Haiku (`claude-haiku-4-5`, $1/$5) for deterministic parse/repair/derivation; Sonnet (`claude-sonnet-4-6`, $3/$15) as the chat/design default; Opus (`claude-opus-4-7`, $5/$25) only when the user opts into max quality.
- **Observability:** token usage (input / output / `cache_read` / `cache_creation`) per call is recorded in the AI-09 in-app log (`bb_api_log`) — the app's cost/latency dashboard (see §7). `max_tokens` caps bound the worst case per call.

## 5. Evaluation Strategy

> **Evaluation philosophy for this phase.** This is a static, no-build, no-backend, single-user BYOK browser app (vanilla ES6 IIFE modules — see `CLAUDE.md`). There is no Python, no server, no CI test runner for AI behavior, and no standing eval service. The project's *established* evaluation mechanism is the GSD **UAT + VALIDATION.md** flow used in Phases 3–6 (user-run, browser-served via `python3 -m http.server`). The eval strategy therefore splits into two layers: (a) **pure-code assertions that run in the browser / `node:test` with zero toolchain** for the deterministic dimensions (schema validity, inventory fidelity, parse success, write-safety gate present), and (b) a **documented reference-prompt set checked via UAT** for the subjective dimensions (balance, technique, voice, personalization), with an optional LLM-judge calibrated against the Section 1b rubric. LLM-judge is reserved only for dimensions that genuinely require it.

### Dimensions

Grounded in Section 1's Critical Failure Modes and the Section 1b expert rubric. "Code" = deterministic check runnable in-browser or via `node:test` against `schema/*.json` and the loaded `State` data (no external tooling). "LLM Judge" = a Claude judge prompt calibrated on the 1b gold examples. "Human" = the home-bar hobbyist (the user) via UAT, optionally a pro bartender for calibration.

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| **Schema validity of generated/parsed JSON** (recipes, drafts, import payloads, JSON-repair output) | PASS: output parses as JSON AND validates against the relevant `schema/*.json` (recipes / drafts / inventory / bar-owner-profile / barkeeper) — all required fields present, correct shape, renders as a chip. FAIL: unparseable, missing a required field, or wrong shape so it breaks the chip render or the recommender engine (FM #2). | **Code** — `JSON.parse` + a lightweight schema-shape validator over `schema/*.json`; runnable as a `node:test` against fixture outputs and live before any write. | **Critical** |
| **Data-write safety** (no write without diff-preview + explicit confirm; invalid schema blocked) | PASS: every AI-driven GitHub write (AI-03 draft, AI-08 import, AI-10 repair) shows a diff preview AND requires explicit user confirmation, AND the payload passed schema validation before the write is offered. FAIL: any code path commits an AI payload with no diff/confirm, or offers to write a schema-invalid payload (FM #1 — silent data corruption). | **Code** — UAT/`node:test` assertion that the write path is gated; structurally verifiable (the confirm + validate step exists in the call chain). | **Critical** |
| **Inventory fidelity / no phantom ingredients + vetoes honored** | PASS: every ingredient in a generated/recommended drink is in the user's inventory (or is explicitly flagged as a *substitution*, not invented as owned); no veto'd ingredient is used un-flagged. FAIL: "phantom bottle" — uses spirits/liqueurs the user does not own, or silently uses a veto'd/disliked ingredient (FM #3; 1b Buildability row). | **Code** — set-membership check of recipe ingredients against loaded `inventory.json` tokens + veto list (the recommender already normalizes inventory tokens; reuse). Edge cases via Human/UAT. | **Critical / High** |
| **Cocktail balance & technique correctness** | 1-5: 5 = no axis dominates, lands ~20–25% water dilution, drinkable; correct method (shake citrus/juice/dairy/egg, dry-shake egg first; stir spirit-forward); ratios hold (sour ≈2:1:1, Negroni ≈1:1:1, Old Fashioned spirit-forward). 1 = cloying/harsh/flabby, or shakes a Manhattan / stirs a sour, or off-template specs no bartender would pour (1b balance/technique/ratio rows). PASS ≥ 4. | **LLM Judge** (Claude judge calibrated on 1b gold "good vs. flagged" examples) **+ Human** for final "would I make and drink this?" | **High** |
| **Context faithfulness / personalization** | PASS: response uses the supplied shared context (flavor profile, mood, made-history, inventory, persona) and does not contradict the locked bar-owner profile; suggestions sound like *this* user's bartender. FAIL: a generic crowd-pleaser ignoring the locked profile/mood/history, or a statement that contradicts the profile (1b "generic, non-personalized" failure mode). | **LLM Judge + Human** (UAT: "did Bjorn use my actual bar/tastes?") | **High** |
| **Structured-output parse success rate** | PASS: a JSON-only feature returns valid parseable JSON on the first attempt, or after exactly ONE auto-retry; the one-retry-then-fail-closed contract holds (no write on failure). FAIL: requires >1 retry, returns prose mixed with JSON, or a parse failure that is not surfaced. Target: ≥ 95% first-or-one-retry success across the reference set. | **Code** — count parse outcomes over the reference set + the live AI-09 call log; deterministic. | **High** |
| **Persona / voice consistency** | 1-5: sounds like the configured bartender persona (`barkeeper.json` name/voice/attitude), classroom-style teaching when asked, mixologist "why it works" depth. PASS ≥ 4. | **LLM Judge + Human** (subjective; calibrate judge on a few persona-matched samples). | **Medium** |
| **Responsible-consumption / allergen handling** | PASS: surfaces a light caveat on overproof or high-volume builds; flags common allergens (egg white, dairy, nuts) and honors user allergies/vetoes; no medical/health framing of alcohol; assumes of-age audience (1b soft editorial guardrails). FAIL: large/overproof pour or risky prep with no caveat, un-flagged allergen, or a health claim. | **LLM Judge + Human** (sample real outputs; not a hard block — editorial guardrail). | **Medium** |

### Eval Tooling

**Primary Tool: GSD UAT + VALIDATION.md (manual, browser-served) + `node:test` pure-code assertions.** This **overrides the agent's Arize Phoenix / RAGAS / Langfuse default** — see rationale below. Those tools assume a Python/Node backend, an OpenTelemetry collector, or a standing eval service that this no-backend static SPA does not have and will not add (it would violate the locked no-build ethos in Section 2 / `CLAUDE.md`). The deterministic dimensions are validated exactly the way Phase 6 validated its engine: `node:test` + `node:assert/strict` loading the IIFE modules via `vm.runInThisContext` with stubbed browser globals (no install). The subjective dimensions are validated by the user via a phase UAT keyed to the reference set.

**Override note:** Arize Phoenix is explicitly NOT adopted as the eval/tracing backend. Production observability is instead the in-app **AI-09 API call log** (`localStorage.bb_api_log`) — see Section 7. Phoenix/promptfoo/Langfuse may be run *optionally* by a developer locally during prompt iteration, but they require Node and are **not a project default or dependency**.

**Setup:**
```bash
# Zero-install. Serve the app the same way Phases 3-6 did, then run UAT in the browser:
python3 -m http.server 8000          # open http://localhost:8000/app/

# Deterministic checks reuse the existing node:test harness (no npm, no config):
node tests/phase-07-ai.test.js       # schema-shape, inventory-fidelity, parse-success fixtures
```

**CI/CD Integration:**
```bash
# Same convention as 06-VALIDATION.md — runs and exits, no watch mode, <1s, no AI calls:
for f in tests/*.test.js; do node "$f" || exit 1; done
# NOTE: CI runs ONLY the deterministic, offline assertions (schema/shape/fidelity/parse over
# recorded fixtures). It makes NO live Anthropic calls (BYOK key never lives in CI). The
# subjective dimensions (balance/technique/voice/personalization) are gated by the user-run
# 07-UAT.md before /gsd-verify-work, exactly as in Phase 6.
```

**Optional dev-time tooling (NOT a default, requires Node):** a developer iterating on prompts may locally run `promptfoo` against the reference set captured as a YAML config to spot prompt regressions. This is convenience-only and is never wired into the app or CI.

### Reference Dataset

**Size:** ~15–20 examples to start (matches the 06-UAT scale; lightweight and browser/manual-runnable — no Python harness). Created *during* implementation alongside each feature, not after.

**Composition:** representative inputs paired with expected properties, covering critical paths, edge cases, and the Section 1/1b failure modes:
- **Chat / quick-ask questions** (persisted `#chat` and ephemeral drawer) — e.g. "what should I make tonight?", a classroom "why do we stir a Manhattan?" — checked for personalization + voice + technique correctness.
- **Recipe-design prompts paired with a known inventory** — e.g. "design me a spirit-forward whiskey drink" against a fixture `inventory.json` — checked for buildability (inventory fidelity), balance, ratio, and schema validity of the generated draft.
- **A malformed JSON blob for repair** (AI-10) — checked that repair output validates against schema and only writes after diff + confirm.
- **A legacy markdown blob for import** (AI-08) — checked it parses into the correct `schema/*.json` shapes with a diff preview.
- **Ambiguous paste-a-line entries** (AI-11) — checked the Claude fallback parses an entry the regex parser cannot.
- **Edge / adversarial cases:** empty inventory (must not invent bottles); user requests a **veto'd spirit** (must flag, not silently use); an **allergen** request like egg white (must surface caveat); an **out-of-template ratio** request (e.g. "an Old Fashioned with orange juice" — must push back or flag, not produce nonsense); an overproof / high-volume request (responsible-consumption caveat); a prompt asking for fabricated provenance of a "classic" (must not invent history).

**Labeling:** the **home-bar hobbyist (the user)** is the primary labeler via UAT — final pass/fail on "would I actually make and drink this?" and "did Bjorn use my real bar?" (per Section 1b expert roles). The **balance / technique / ratio** rows may additionally use a **Claude LLM judge calibrated against the 1b gold examples** (and optionally a pro-bartender calibration of a handful of "good vs. flagged" reference drinks). Deterministic rows (schema, inventory fidelity, parse success, write-safety) are auto-labeled by the `node:test` assertions.

## 6. Guardrails

All online guardrails run **client-side in the browser** (no server). They favor *fail-closed* on the data-integrity path: when in doubt, do not write.

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **No-API-key gate** (CHAT-03) | Any AI surface opened with no `bb_anthropic_key` present | **Block** — show "No API key" message linking to Settings; no call attempted. |
| **Schema validation before any GitHub write** | Before every AI-driven write (AI-03 draft, AI-08 import, AI-10 repair) | **Block** — payload that fails its `schema/*.json` shape check is never offered for write; surface the validation error (FM #1/#2). |
| **Diff preview + explicit user confirm before every write** (AI-03 / AI-08 / AI-10) | Any AI payload about to be committed to GitHub | **Block until confirmed** — render a diff of old vs. new and require an explicit user confirm; no silent commit. |
| **JSON-only output contract — one auto-retry then fail-closed** | A JSON-only feature returns unparseable / non-conforming output | **Flag + retry once; then Block** — exactly one auto-retry; on second failure, surface the error and do NOT write (no partial/garbage commit). |
| **Inventory-fidelity check on generated recipes** | A generated/recommended drink references an ingredient not in inventory and not flagged as a substitution | **Flag** — mark phantom ingredients in the draft preview so the user sees them before promoting to Original (FM #3). |
| **AbortController on navigation** (CHAT-06) | User closes the drawer / leaves `#chat` mid-stream | **Block (abort)** — `cleanup()` aborts the in-flight stream; persisted transcript unaffected; avoids orphaned calls/spend. |
| **429 rate-limit handling** (CHAT-09) | Anthropic returns HTTP 429 | **Flag, do not retry-storm** — surface the `retry-after` value to the user; no automatic retry loop. |
| **Mid-stream SSE error surfacing** (CHAT-08) | An `error` event arrives in the SSE stream | **Flag** — render a user-readable error message in the conversation; abort cleanly. |
| **max_tokens cap per call** | Every Messages API call | **Block (cap)** — a `max_tokens` ceiling bounds output length and per-call cost on the user's BYOK key (FM #5). |
| **Key non-exfiltration invariant** | Every call / log write | **Block** — key only sent to `api.anthropic.com` with `anthropic-dangerous-direct-browser-access: true`; key is NEVER written to `bb_api_log`, never committed to GitHub, masked in the UI (FM #4). |

### Offline (Flywheel)

These are *user-driven*, signal-based reviews (no automated batch job — single-user app). They feed prompt/feature improvement, not paging.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Logged parse failures / invalid outputs** | Retain failed/invalid responses surfaced in the AI-09 call log (`bb_api_log`); user reviews periodically | If a feature shows repeated parse failures, revise its JSON-only prompt / few-shot examples before adding retries. |
| **Generated-recipe acceptance rate** (promote-to-Original vs. reject) | Track which AI drafts the user promotes to Original vs. deletes — a direct quality signal for balance/buildability | Low promotion rate → re-tune the recipe-design system prompt against the 1b balance/ratio/inventory rubric. |
| **Phantom-ingredient incidence** | Sample drafts where the inventory-fidelity flag fired | Recurring phantoms → strengthen the inventory/veto block in the context builder and the design prompt. |
| **Periodic AI-09 call-log review** | User scans the 50-entry log for anomalies (cost spikes, 429 clusters, low cache-hit rate) | Adjust model selection (SET-05), prompt-caching granularity, or call frequency. |

## 7. Production Monitoring

**Tracing Tool: the in-app AI-09 API call log (`localStorage.bb_api_log`).** This is the **default/primary observability surface for this app, and it overrides the agent's Arize Phoenix default** because there is no backend, no OpenTelemetry collector, and no standing service to send traces to — the app runs entirely in the user's browser. Every Anthropic call is recorded (50-entry rolling cap) with timestamp / type (chat, recipe-design, import, repair, etc.) / model / token usage; a Settings panel renders the log with copy-raw-JSON and clear-log actions. The key is never written to the log (FM #4). Optional local dev tools (`promptfoo`, Phoenix, Langfuse) are **not the default** and require Node; a developer may run them locally during prompt iteration only.

**Key Metrics to Track:**
- **Call count + estimated cost** — number of calls in the log window and rough spend on the user's BYOK key.
- **Token usage** — input / output / cache-read / cache-write tokens, to watch the **prompt-cache hit rate** (D-05 caching is what makes the rich context affordable).
- **JSON parse-failure rate** — share of JSON-only calls that failed to parse on first attempt (or needed the one retry).
- **429 rate** — frequency of rate-limit responses.
- **Generated-recipe acceptance (promote-to-Original) rate** — quality signal for the recipe-design feature.

**Alert Thresholds:** This is a **client-side single-user app — there is no paging and no alerting infrastructure** (stated explicitly). Instead, monitoring surfaces **warnings in the Settings AI-09 panel** when a signal looks off, for example: a **low prompt-cache hit rate** (suggests the cache is being invalidated too often / context churn), a **spike in JSON parse failures** for a feature, or a **cluster of 429s**. These are advisory nudges for the user/developer, not automated alerts.

**Smart Sampling Strategy:** **Signal-based retention, not random sampling.** The call log preferentially retains and flags **failed / invalid responses** and **rejected generations** (AI drafts the user deleted rather than promoted) so they are available for user review and prompt tuning. Successful, accepted interactions need no review; the limited 50-entry budget is spent on the outliers that carry the most improvement signal.
